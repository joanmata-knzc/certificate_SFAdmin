Security & Access

Seguridad y Control de Accesos

El sistema de capas que define quién puede ver qué y hacer qué dentro de Salesforce.

🏛️ La Pirámide de Seguridad (de arriba hacia abajo):
1. Perfil / Permission Sets → Controlan QUÉ ACCIONES puede hacer el usuario.
2. OWD (Org-Wide Defaults) → Define la visibilidad mínima por defecto de los registros.
3. Jerarquía de Roles → Los jefes ven lo de sus subordinados automáticamente.
4. Sharing Rules → Amplían el acceso a grupos específicos de forma automática.
5. Manual Sharing → Un usuario comparte un registro concreto con otro usuario.
La seguridad siempre es lo más restrictivo que define el Perfil/FLS y lo más abierto que permite la cadena de sharing.

1. Perfiles (Profiles) — ¿Qué puede HACER?

Dónde: Setup → Users → Profiles
Cada usuario tiene exactamente UN solo perfil. El perfil define los permisos base.

🎯 Object Permissions (CRUD)

  • Create: Puede crear registros nuevos.
  • Read: Puede ver registros.
  • Edit: Puede modificar registros existentes.
  • Delete: Puede borrar registros.
  • View All / Modify All: Ignora las reglas de sharing (solo para admins).

⚙️ Otras Configuraciones del Perfil

  • App Access: Qué aplicaciones ve el usuario en el App Launcher.
  • Tab Settings: Si una pestaña está visible, oculta o predeterminada.
  • Field-Level Security (FLS): Para cada campo, si el usuario puede verlo y/o editarlo.
  • Login Hours / IP Ranges: Restricción de acceso por horario y red.
  • Password Policies: Caducidad, complejidad de contraseña.

2. Permission Sets y Permission Set Groups

Dónde: Setup → Users → Permission Sets

🔑 Permission Sets

Amplían los permisos de un perfil sin crear un perfil nuevo. Un usuario puede tener múltiples Permission Sets.

Ejemplo: 50 usuarios tienen el perfil "Ventas", pero 3 de ellos también gestionan contratos. Les asignas un Permission Set de "Contratos" solo a esos 3.

📦 Permission Set Groups

Agrupan varios Permission Sets en un paquete para asignarlos de una sola vez. Útil cuando los roles son complejos.

Muting Permission Sets: Dentro de un grupo, puedes bloquear ciertos permisos específicos con un "Muting Permission Set".

💡 Tip de Examen: Los Permission Sets solo pueden ampliar permisos, nunca restringirlos. Para restringir, debes modificar el Perfil o usar FLS.

3. OWD — Organization-Wide Defaults (¿Qué REGISTROS ve?)

Dónde: Setup → Security → Sharing Settings
Define el nivel de acceso mínimo por defecto a los registros que no son de tu propiedad.

Nivel OWD ¿Qué puede hacer alguien que NO es dueño? Caso de uso
Private Nada. Solo el dueño y sus superiores en el rol lo ven. Datos sensibles: expedientes de RRHH, oportunidades confidenciales.
Public Read Only Ver el registro pero no editarlo. Catálogo de productos, base de conocimiento interna.
Public Read/Write Ver y editar. Contactos compartidos entre toda la empresa.
Public Read/Write/Transfer Ver, editar y reasignar el dueño. Leads en una org muy colaborativa.
Controlled by Parent El acceso al hijo depende del acceso al padre (para Master-Detail). Líneas de pedido que siguen la seguridad del pedido.
💡 Tip de Examen: OWD solo puede restringir. Para ampliar el acceso a registros específicos se usan los mecanismos siguientes (Roles, Sharing Rules, Manual Sharing).

4. Jerarquía de Roles y Sharing Rules

🌳 Role Hierarchy (Jerarquía de Roles)

  • Es un árbol organizativo. Los usuarios en roles superiores ven automáticamente los registros de sus subordinados.
  • No tiene nada que ver con los Perfiles (los perfiles controlan acciones, los roles controlan visibilidad de registros).
  • Se puede desactivar por objeto marcando "Grant Access Using Hierarchies" como desactivado.

🔗 Sharing Rules

Excepciones automáticas al OWD. Amplían el acceso de ciertos registros a ciertos grupos.

  • Owner-Based: Comparte registros cuyo dueño pertenece a un grupo/rol.
  • Criteria-Based: Comparte registros que cumplen ciertos criterios de campo (ej: Country = "Spain").

Solo pueden ampliar el acceso, nunca restringirlo más allá del OWD.

5. Field-Level Security (FLS) — Seguridad a nivel de campo

🔒 ¿Qué es?

Controla si un usuario puede ver y/o editar un campo específico, incluso si tiene acceso al registro.

  • Se configura en el Perfil o Permission Set, campo por campo.
  • Si FLS dice "oculto", el usuario no verá ese campo en ningún sitio: ni en la pantalla, ni en reportes, ni en búsquedas.
  • El Page Layout puede añadir campos al diseño, pero si FLS los oculta, el campo sigue sin verse.
🔑 Regla de oro FLS vs Page Layout:
El Page Layout controla el diseño visual. El FLS controla la seguridad real.
Si hay conflicto, el FLS siempre gana. Un campo marcado como "Read Only" en FLS será de solo lectura aunque el layout diga que es editable.
