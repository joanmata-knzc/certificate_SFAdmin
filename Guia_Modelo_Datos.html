Object Manager & Lightning App Builder

Modelo de Datos y Personalización de UI

Aquí defines qué información guarda Salesforce y cómo la ve el usuario.

1. Objetos: Las Tablas de la Base de Datos

Dónde: Setup → Object Manager (o la barra de búsqueda de Setup)

📋 Standard Objects (Objetos Estándar)

  • Vienen incluidos con Salesforce. No se pueden borrar.
  • Ejemplos clave: Account (empresa), Contact (persona), Lead (candidato), Opportunity (venta), Case (soporte).
  • Se pueden personalizar añadiendo campos o layouts.

🛠️ Custom Objects (Objetos Personalizados)

  • Los creas tú para necesidades específicas del negocio.
  • Su API name siempre termina en __c (ej: Proyecto__c).
  • Pueden tener sus propias relaciones, layouts y automatizaciones.
  • Al crearlos, Salesforce genera automáticamente campos de sistema (Created By, Modified By...).
💡 Tip de Examen: Los campos y objetos personalizados llevan el sufijo __c. Los campos de relación llevan __r (para referencias). Esto suele aparecer en preguntas de fórmulas.

2. Fields & Relationships (Campos y Relaciones)

Dónde: Object Manager → [Selecciona un objeto] → Fields & Relationships

Tipos de Relaciones

Tipo Descripción ¿Qué pasa si se borra el padre? ¿Roll-up?
Lookup Enlace "suave". El campo es opcional. No hereda seguridad. El hijo sobrevive (el campo queda vacío). ❌ No
Master-Detail Enlace "fuerte". El hijo depende del padre. Hereda su seguridad. Los hijos se borran también (cascada). ✅ Sí
Many-to-Many Se implementa con un objeto intermedio (Junction Object) con dos Master-Detail. Depende de cuál padre se borra. ✅ En el padre

Tipos de Campos más Importantes

📝 Campos de Entrada de Datos

  • Text: Texto libre (max 255 chars).
  • Text Area (Long/Rich): Texto largo, incluso con HTML.
  • Number / Currency / Percent: Valores numéricos con formato.
  • Date / Date-Time: Fechas y timestamps.
  • Checkbox: Verdadero/Falso.
  • Picklist: Menú desplegable con opciones predefinidas. Garantiza calidad del dato.
  • Multi-Select Picklist: Permite seleccionar varias opciones a la vez.
  • Lookup / Master-Detail: Campos de relación entre objetos.

⚙️ Campos Calculados / Especiales

  • Formula: Solo lectura. Calcula su valor automáticamente. Puede usar campos de otros objetos relacionados.
  • Roll-up Summary: Solo en Master-Detail. Calcula COUNT, SUM, MIN o MAX de los registros hijos en el padre.
  • Auto-Number: Genera un identificador automático incremental (ej: CASO-0001).
  • External ID: Campo único que sirve de "puente" con sistemas externos (ej: ID del ERP). Útil en cargas de datos.
💡 Tip de Examen: Roll-up Summary solo funciona en relaciones Master-Detail, nunca en Lookup. Si preguntan cómo calcular la suma de los hijos en el padre en una relación Lookup, la respuesta es usar un Flow.

3. Record Types (Tipos de Registro)

Dónde: Object Manager → [Objeto] → Record Types

🗂️ ¿Para qué sirven?

Permiten que un mismo objeto tenga diferentes procesos de venta, opciones de picklist y page layouts según el perfil del usuario o el tipo de negocio.

Ejemplo práctico: En el objeto Account, puedes tener:

  • Record Type "Cliente": Muestra campos de facturación, con un layout específico para el equipo de ventas.
  • Record Type "Partner": Muestra campos de contrato, con un layout para el equipo de alianzas.

Los Record Types se asignan a los Perfiles. Un perfil puede tener acceso a uno o varios Record Types.

4. Page Layouts y Lightning App Builder

📄 Page Layouts

Controlan la organización de los campos, secciones, listas relacionadas y botones en la vista de un registro.

  • Se asignan a combinaciones de Perfil + Record Type.
  • Puedes hacer un campo obligatorio en el layout (aunque no lo sea en el objeto).
  • Puedes marcar un campo como solo lectura en el layout.
  • Ojo: Si el FLS del perfil dice que un campo es de solo lectura, gana el FLS sobre el layout.

⚡ Lightning App Builder

Editor drag-and-drop para construir páginas modernas con componentes.

  • App Pages: Páginas de inicio de una aplicación.
  • Home Pages: La pantalla de inicio de los usuarios.
  • Record Pages: Vista de un registro. Reemplaza al Page Layout clásico en Lightning.

Dónde: Setup → User Interface → Lightning App Builder

Las páginas se activan como predeterminadas para toda la org, para una app, o para perfiles específicos.

🔑 Relación clave Page Layout vs Lightning Record Page:
El Page Layout define qué campos están disponibles en la página del registro.
La Lightning Record Page define cómo están organizados los componentes (dónde va el panel de actividad, dónde van los campos, etc.).
Ambos trabajan juntos: el Record Page usa el Page Layout para el componente de detalle del registro.
